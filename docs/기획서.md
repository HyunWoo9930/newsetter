# 🧗 NewSetter — 클라이밍 크루 암장 방문 트래킹 & 투표 서비스

## 1. 한 줄 소개
"우리 크루, 이 암장 뉴셋 이후로 갔었나?" — 벽 세팅 주기 기준으로 크루의 암장 방문을 추적하고, 웬투밋처럼 투표로 다음 갈 곳을 정하는 서비스.

## 2. 문제 정의 (실제 페인포인트)
1. **암장 벽은 보통 4주마다 전체 리셋(뉴셋)되는데**, 우리 크루가 "이번 세팅에서" 그 암장을 갔는지 아닌지 확인하려면 카톡 기록을 뒤져야 한다.
2. 다음 주에 어디 갈지 정할 때 카톡 투표로 하는데, 암장+날짜+시간을 한 번에 조율하기 번거롭다 (웬투밋 따로, 카톡 투표 따로).
3. 안 가본 암장은 정보가 부족하다 — 다른 크루/유저의 리뷰를 보고 결정하고 싶다.

## 3. 타겟 사용자
- **주 타겟**: 이미 활동 중인 클라이밍 크루 (크루장 + 크루원)
- **부 타겟**: 새로 크루를 만들려는 클라이머, 암장 선택에 참고할 리뷰가 필요한 개인 클라이머

## 4. 핵심 기능 (MVP)

### 4.1 ⭐ 방문 트래킹 — "언제 어디 갔는지" (킬러 기능 1)
- 크루 캘린더/타임라인: 우리 크루가 언제 어떤 암장을 방문했는지 기록
- **암장별 세팅 주기 기반 알림**: 암장마다 뉴셋 주기(기본 4주)와 최근 세팅일을 등록해두면
  - "이 암장, 뉴셋 이후 아직 안 감" / "이번 세팅에서 이미 방문" 상태를 자동 표시
  - 암장 목록이 "가야 할 곳(뉴셋 됐는데 안 간 곳)" 순으로 정렬됨
- 방문 기록은 투표로 확정된 일정에서 자동 생성 + 수동 추가도 가능

### 4.2 ⭐ 일정 투표 (킬러 기능 2)
- 크루장이(또는 아무나) 투표 생성: **날짜 후보 여러 개 + 암장 후보 여러 개**
- 시간대 그리드 없이 **날짜(요일+시간대 라벨) 단위 투표** (예: "화 저녁 / 목 저녁 / 토 오후")
- **응답 순서: ① 가능한 날짜 선택 → ② 선호 암장 선택** (언제 갈지가 먼저 정해져야 어디 갈지 고르는 흐름)
- 암장 후보 옆에 "뉴셋 D+며칠 / 우리 크루 방문 여부 / 평점"이 함께 표시되어 투표에 참고
- 마감 시 최다 득표 조합(날짜+암장)이 일정으로 확정 → 크루 캘린더에 등록 → 방문 기록으로 연결

### 4.3 ⭐ 암장 리뷰 (킬러 기능 3)
- 암장별 리뷰: 별점 + 태그(세팅 퀄리티, 혼잡도, 초보 친화, 주차, 샤워시설 등) + 텍스트
- **"이번 세팅" 단위 리뷰**: 같은 암장이라도 세팅마다 평가가 달라지므로, 리뷰가 세팅 회차에 묶임 ("이번 셋 재밌다/노잼")
- 크루 단위 리뷰도 가능 ("○○크루가 다녀감 — 만족도 ★4")
- 투표 생성 시 후보 암장의 최근 리뷰가 바로 보임

### 4.4 뉴셋 알림 & 뉴셋 공지 링크
- **뉴셋 알림**: 관심 암장(즐겨찾기 or 크루 주 활동 암장)의 세팅 주기가 도래하면 알림 (PWA 웹 푸시)
- **뉴셋 공지 링크**: 세팅 회차에 암장 인스타 뉴셋 공지/영상 링크 첨부 — 투표·리뷰 화면에서 미리보기 참고

### 4.5 크루 & 암장 기본 기능
- 크루 생성
- **가입 2가지 경로**:
  - **초대 링크** → 링크로 들어오면 바로 가입 (승인 불필요, 크루장이 링크를 준 사람이므로 신뢰)
  - **가입 신청** → 검색 등으로 찾아와 신청하면 크루장 승인 후 가입 (status: PENDING → APPROVED)
- 암장 DB: 이름, 위치(지도), 세팅 주기, 최근 세팅일 — 유저 제보로 세팅일 갱신(크라우드소싱)
- 개인 프로필: 내 방문 히스토리, 내가 쓴 리뷰

## 5. MVP에서 제외 (v2 이후)
- 채팅 (크루 오픈카톡 링크로 대체)
- 크루 공개 모집/매칭
- 문제(볼더) 단위 개인 기록·난이도 통계
- 시간대 그리드 조율 (날짜 단위 투표로 충분하다고 판단, 필요해지면 v2)

## 6. 화면 구성 (모바일 웹)
1. **크루 홈** — 다가오는 확정 일정, 진행 중인 투표, "가야 할 암장" 리스트(뉴셋 기준 정렬)
2. **투표 생성/참여** — ① 날짜 후보 선택 → ② 암장 후보 선택 (참여자는 가능한 날짜 → 선호 암장 순으로 체크)
3. **크루 캘린더** — 월별 방문 기록, 암장별 필터
4. **암장 상세** — 정보, 세팅 히스토리, 이번 세팅 리뷰, 우리 크루 방문 여부
5. **암장 탐색** — 지도/리스트, 필터(지역, 뉴셋 임박, 평점)
6. **프로필** — 내 방문/리뷰

## 8. 기술 스택 (혼자 바이브 코딩 + 자체 Oracle 서버 운영)
- **Next.js (App Router) + TypeScript + Tailwind CSS** — 프론트+백엔드 한 몸, 모바일 웹 최적, PWA(웹 푸시) 지원
- **PostgreSQL** — Docker 컨테이너로 구동, Prisma ORM으로 접근
- **인증**: NextAuth(Auth.js) + 카카오 OAuth
- **푸시**: web-push (VAPID) — 뉴셋 알림용, 별도 유료 서비스 불필요
- **Kakao Map API** — 암장 위치/탐색
- **배포**: 로컬에서 개발 완료 → 보유 중인 Oracle 서버에 Docker Compose(Next.js + Postgres + Caddy)로 배포
  - Caddy로 HTTPS 자동 발급 (PWA 푸시와 카카오 로그인 모두 HTTPS 필수)

## 8.1 배포 전략
1. **로컬 개발**: `docker compose up`으로 Postgres만 띄우고 Next.js는 dev 서버로 개발
2. **기능 완성 후**: 같은 compose 파일에 Next.js 빌드 이미지 + Caddy 추가해서 Oracle 서버에 배포
3. 도메인 1개 연결 + Caddy가 인증서 자동 처리 → 카카오 개발자 콘솔에 redirect URI 등록

## 12. (확장) 벽 문제 분석 & 문제별 커뮤니티

### 목표
뉴셋이 되면 그 세팅의 문제 목록(색=난이도별)이 만들어지고, 각 문제마다
"꿀인지 / 어려운지 / 베타(꿀팁)"를 모으는 커뮤니티.

### ⭐ 실제 유저 페인 → 해결 원리 (난이도 정규화)
유저 페인은 두 가지:
1. **다른 암장 가면 난이도 감이 안 옴** — 내 홈짐 빨강이 여기선 무슨 색인지 모름.
2. **같은 난이도 안에서도 쉬운 문제/어려운 문제가 섞임** — 많이 풀려면 쉬운 것부터 해야 하는데 뭐가 쉬운지 알 방법이 없음.

둘 다 **크라우드소싱된 "체감 난이도" 하나로 풀린다. AI 불필요.** 핵심은 두 층위로 나누는 것:

- **(A) 색 단위 보정 — 안정적, 세트를 넘어 영구 누적 → 페인 1 해결**
  - 암장의 색→난이도 매핑은 뉴셋돼도 대체로 유지됨(그 암장 빨강은 늘 비슷한 등급대).
  - 그래서 "이 암장의 이 색 ≈ 공통 척도(V등급) 몇" 을 유저 투표로 계속 쌓음.
  - 유저 프로필에 홈짐+편한 등급을 기준으로 두면 → "여기 빨강 = 너의 홈 파랑 수준" 자동 환산.
  - **크루 앱이라 크루가 여러 암장을 돌아다님 = 암장 간 난이도를 잇는 다리(bridge) 데이터가 자연히 쌓임.**
- **(B) 문제 단위 순위 — 세트 내에서 빠르게 누적 → 페인 2 해결**
  - 같은 색 안에서 문제별 체감 난이도 점수를 매겨 **쉬운 순으로 정렬.**
  - 신호: 체감 투표(쉬웠/적정/어려웠), 완등률, 시도 횟수, "꿀" 플래그.
  - "빨강 중 이 3개가 제일 쉬움 → 워밍업·다작용" 을 바로 보여줌.

### 데이터가 없을 때(콜드스타트) 우아한 폴백
- 리뷰가 없으면 **색=난이도의 거친 순서**로만 정렬(색 자체가 이미 등급). 데이터가 쌓일수록 색 안에서 세밀해짐.
- 새 암장/새 세트는 데이터가 없으니, **입력 마찰을 최소화**해야 함: 방문 기록 후 "방금 푼 문제 어땠어요?" 한 탭(쉬움/적정/어려움 + 완등 여부) 프롬프트. 크루라는 소규모 신뢰 그룹이 초기 데이터를 채우는 구조가 유리.

### 난이도 점수 계산 (MVP → 고도화)
- **MVP**: 문제 점수 = 체감 투표 가중 평균(쉬움 -1 / 적정 0 / 어려움 +1) + 완등률 보정. 단순 집계로 충분.
- **v2 고도화**: "A가 B보다 어려웠다" 쌍대 비교를 모아 Elo/Bradley-Terry로 연속 점수화 → 더 견고. 색 보정도 같은 방식으로 암장 간 상대비교 가능.

### ⭐ 영상 기반 입력 = 이 커뮤니티의 엔진
난이도 데이터를 "폼 채우기"로 받으면 아무도 안 함. 그런데 **클라이머는 원래 자기 등반을 찍음.**
그 습관에 올라타서, **영상을 올릴 때 문제 메타데이터(난이도/완등/체감/꿀)를 같이 받는다.**
→ 영상 업로드가 곧 난이도 데이터 입력이 됨. 위 (A)(B) 계산의 연료가 여기서 나옴.

- **영상 = 최고의 베타.** 텍스트 꿀팁보다 "그 사람이 어떻게 풀었는지" 영상 한 개가 훨씬 값짐.
- **한 방문에 여러 개**: "이번에 뭐 풀었어?" → 문제별로 영상+메타를 여러 개 추가(요청하신 "영상 7개" 시나리오).
- **영상은 선택**. 핵심 신호는 메타데이터(완등/체감/꿀)라, 영상 없이 원탭 기록도 가능해야 함(마찰 최소화). 영상은 있으면 풍부해지는 보너스.

### 냉정한 평가 & 리스크
- **좋은 이유**: 입력 마찰/콜드스타트 문제를 정면 해결. 로그(내 완등 기록) + 크루 베타 라이브러리 + 난이도 데이터가 한 번의 업로드로 동시에 쌓임.
- **가장 큰 리스크 = 영상 호스팅 비용/운영.** 저장·대역폭·트랜스코딩·재생. Oracle 서버에 직접 얹으면 **egress(전송량)가 킬러**.
  - 권장: **Cloudflare R2**(전송 비용 0) 또는 **Cloudflare Stream**(트랜스코딩+적응형 재생 내장, 분당 과금). 클립이 짧아(10~40초) 비용 통제 쉬움.
  - 더 가벼운 시작: 유튜브/인스타 **링크 임베드** → 저장 0원. 대신 UX 파편화.
  - **Oracle 서버에 원본 비디오를 primary로 두지 말 것.**
- **"다들 올릴까?" — 여기가 진짜 관건.** 인스타/유튜브와 조회수로 경쟁하면 짐. 그래서 포지셔닝을 다르게:
  낯선 사람 좋아요가 아니라 **① 내 등반 로그 ② 우리 크루 베타 공유 ③ 내가 가는 암장 난이도 지도.**
  = 조회수가 아니라 **효용**이 동력. 소규모 크루에서도 성립.

### 핵심 통찰: 두 개의 문제를 분리한다
1. **문제별 커뮤니티(쉬움, 가치 큼)** — 문제 목록만 있으면 리뷰·베타 CRUD는 지금 구조의 확장.
   이미 있는 `GymSetting`(세팅 회차)에 `Problem`을 매달면 "이번 셋" 개념과 딱 맞음.
2. **사진으로 벽 자동 분석(어려움, 리스크)** — OCR 아님. **컴퓨터 비전**(홀드 객체 탐지 + 색 분류 + 색→난이도 매핑).
   → 이건 커뮤니티의 "입력 편의 기능"일 뿐. 없어도 커뮤니티는 돌아감. 그래서 **나중에, 선택적으로.**

### 자동 분석을 꼭 "벽 사진"으로 할 필요는 없다
- 많은 암장이 뉴셋 때 **세팅표(문제 목록 이미지/공지)를 인스타·게시판에 올림.**
  → 벽에서 홀드를 탐지하는 것보다 **그 세팅표 이미지를 VLM/OCR로 읽는 게 훨씬 쉽고 정확.**
- 우리는 이미 세팅 회차에 "뉴셋 공지 링크"를 붙이게 해놨음 → 그 이미지를 파싱하는 흐름이 자연스러움.

### 접근법 비교 (벽 사진 직접 분석)
| 방법 | 정확도 | 공수 | 비고 |
|---|---|---|---|
| A. VLM 제로샷 (사진→Claude/GPT vision) | 중 (색 목록 O, 정확한 개수 X) | 낮음 | 프로토타입 즉시. "빨강·파랑 문제 있음" 수준 |
| B. 커스텀 홀드 탐지기 (YOLO + Roboflow 데이터셋) + 색 클러스터링 | 높음 | 매우 높음 | GPU 추론 호스팅 필요, 혼자선 부담 |
| C. 크라우드소싱 우선 (사람이 최초 1회 입력, 이후 리뷰) | 높음 | 낮음 | AI 보조는 옵션. **MVP로 최적** |

### 왜 벽 자동 분석이 어려운가 (현실 체크)
- 홀드는 모양·크기·볼륨·조명·가림이 제각각 → 범용 모델이 "클라이밍 홀드"를 모름. 커스텀 학습 필요.
- 색=난이도지만 **색→등급 매핑이 암장마다 다름**, 같은 색 문제가 한 벽에 여러 개일 수 있음, 한 장에 벽 전체가 안 담김.
- 결국 "정확한 문제 개수"를 사진 한 장으로 자동 확정하긴 어려움 → 사람 확인(confirm) 단계가 사실상 필수.

### 추천 단계 (phased)
- **12a (v2, 쉬움)**: `Problem`(세팅 회차에 소속: 색/등급/라벨) 수동·크라우드소싱 생성 + **문제별 베타·체감난이도·꿀지수 리뷰**. ← 진짜 가치, 여기부터.
- **12b (v2.5)**: **VLM 보조 입력** — 세팅표 or 벽 사진을 올리면 색/등급 목록을 *제안*, 사람이 확인·수정.
- **12c (v3, 선택)**: 수요·데이터가 쌓이면 커스텀 홀드 탐지기 검토.

### 데이터 모델 (초안)
```
Problem (id, gymSettingId, color, grade?, label?, photoUrl?, createdById, createdAt)

-- 페인 2 + 영상 로그: 문제별 완등/체감 기록 (영상은 선택 첨부)
ClimbLog (id, problemId, userId,
          sent(완등여부), attempts?(시도수),
          relativeFeel(EASIER/AS_EXPECTED/HARDER),  -- 그 색 기준 체감
          honey(꿀여부), content(베타/팁),
          videoId?/videoUrl?, videoThumb?,          -- R2/Stream 또는 임베드 링크
          createdAt)
-- 이 한 엔티티가 "완등 로그 + 베타 + 난이도 신호 + 영상"을 겸함.
-- 파생: Problem.difficultyScore(체감 점수), 꿀 비율, 완등률 → 색 내 오름차순 정렬

-- 페인 1: 암장 간 난이도 보정 (색→공통척도)
ColorGradeVote (id, gymId, color, userId, vGrade(공통 V척도 추정), createdAt)
-- 파생: (gymId,color) → 평균 vGrade. "여기 빨강 ≈ V3" / "너의 홈 파랑 ≈ V3 = 여기 빨강"

-- 개인화 기준점 (User 확장)
User += homeGymId?, referenceColor?/referenceGrade?  -- "홈짐에서 편하게 푸는 색"
```
- `Problem`은 `GymSetting`에 묶임 → 뉴셋되면 자연히 새 세트의 문제 셋이 시작(세트 내 순위는 4주마다 리셋).
- `ColorGradeVote`는 `Gym`+색에 묶임 → 세트를 넘어 영구 누적(암장의 색 난이도는 안정적).

### ⭐ 추천 — "뭐부터 풀지" (개인화)
데이터가 쌓이면 "나와 비슷한 유저 기준으로 뭐부터 풀면 좋을지" 추천 가능. 근본적으로 추천 시스템 문제.

- **결정적 제약: 문제는 4주면 사라짐(휘발).** → 문제 하나하나를 학습하는 **고전 item 기반 CF는 부적합**
  (충분히 모으기 전에 문제가 없어지고, 지난 세트 문제는 추천할 대상 자체가 소멸).
- **올바른 프레임: 유저는 영구 · 문제는 특징으로 서술.**
  → **IRT / Elo 모델**: 유저 능력 θ, 문제 난이도 b, **P(완등) = logistic(θ − b).**
  - θ는 세트·암장을 넘어 계속 누적(휘발 문제 문제 해결). 새 문제 b는 색/크라우드 점수로 시드 후 완등이 들어오면 보정.
  - **암장 간 자동 정규화**: A짐 완등도 B짐 완등도 같은 θ를 업데이트 → 페인 1(암장 간 난이도)도 같이 풀림.
- **추천 로직** = 현재 세트에서 (아직 안 깬) ∧ (b ≲ θ, 할 만한) 문제를 **쉬운 순으로**. "꿀" 신호로 재미있는 것 우선.
- **"나와 비슷한 유저"** = θ + 스타일 유사도로 이웃을 찾아 "너랑 비슷한 사람은 이 다음에 이걸 깼어"(러닝 패스).
  크루 구조 + 공유 암장이 유저 간 overlap을 만들어줘서 유사도 계산에 유리.
- **단계**: v1 규칙기반(내 능력 ≈ 안정 완등 등급 → 그 이하 미완등을 쉬운 순) → v2 IRT/Elo → v3 이웃/스타일/행렬분해.
- **리스크**: 콜드스타트(신규 유저·신규 세트 → 색 기반으로 폴백), 데이터 밀도 부족, 필터버블(살짝 위 난이도도 섞는 "성장 모드" 필요).

### 비용 메모
- VLM 사진 분석은 이미지당 API 비용 발생 → 세팅 회차당 1회(최초 입력자만) 호출하고 결과를 저장·공유하면 비용 통제 가능.
- 추천(IRT/Elo)은 외부 API 없이 자체 계산 → 야간 배치로 θ·b 갱신하면 비용 0에 가까움.
